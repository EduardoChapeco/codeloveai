## Tarefa: Diagnóstico — Extensões mostrando 10 mensagens para Admin

### Causa Raiz

O Admin Master tinha sua licença vinculada ao `plan_id: 66ba3bfc-5e29-4fe5-b569-7ee4325af738` (plano "Grátis", 10 msg/dia). Isso aconteceu porque o sistema de auto-onboarding atribuiu o plano gratuito ao admin como qualquer outro usuário.

**Problema 1 — Token CLF1 com limites embutidos:**
O token CLF1 é um JWT assinado que contém `dailyMessages: 10` e `exp` de 30 dias **hardcoded no payload**. A extensão lê esses valores localmente do token, sem consultar o servidor.

**Problema 2 — plan_id apontava para "Grátis":**
Mesmo com `daily_messages: NULL` na row da license, o `validate-license` retornava os dados do plano vinculado (que tinha `daily_message_limit: 10`).

### Correção Aplicada

1. **DB Fix**: Atualizado `licenses` do admin para `plan_id: NULL`, `daily_messages: NULL`, `plan: 'admin_master'`, `expires_at: 2099`
2. **validate-license**: Já tinha bypass para admin (isAdmin = true → dailyMessages: null)
3. **Ação necessária**: O admin precisa **regenerar o token CLF1** (via Dashboard > Perfil ou API) para que o novo token reflita os dados corretos (unlimited, 2099)

### Por que a chave tinha 30 dias?
O `generate-clf-token` usa `expiresMs = 30 * 24 * 60 * 60 * 1000` como default. Para admin, usa `ADMIN_MASTER_EXPIRES_MS` (até 2099). Mas o token existente foi gerado ANTES da lógica de admin bypass existir, então ficou com 30 dias.
